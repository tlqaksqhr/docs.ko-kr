3.  <span data-ttu-id="e0c24-112">모든 지속성 참가자에서 구현된 <xref:System.Activities.Persistence.PersistenceParticipant.CollectValues%2A> 메서드를 호출합니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-112">Invokes the <xref:System.Activities.Persistence.PersistenceParticipant.CollectValues%2A> method implemented by all persistence participants.</span></span>

 <span data-ttu-id="e0c24-126">다음 목록에서는 로드 작업의 여러 단계에서 지속성 하위 시스템이 수행하는 작업을 설명합니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-126">The following list describes the tasks performed by the persistence subsystem in different stages of the Load operation.</span></span> <span data-ttu-id="e0c24-127">지속성 참가자는 네 번째 단계에서 사용됩니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-127">The persistence participants are used in the fourth stage.</span></span> <span data-ttu-id="e0c24-128">세 번째 단계에서 (I/O 작업에도 참가 하는 지 속성 참가자)의 지 속성 I/O 참가자도 사용 됩니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-128">The persistence I/O participants (persistence participants that also participate in I/O operations) are also used in the third stage.</span></span>

3.  <span data-ttu-id="e0c24-131">호출의 <xref:System.Activities.Persistence.PersistenceIOParticipant.BeginOnLoad%2A> 모든 I/O 지 속성 참가자 및 모든 지 속성 참가자 끝나기를 기다립니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-131">Invokes the <xref:System.Activities.Persistence.PersistenceIOParticipant.BeginOnLoad%2A> on all persistence I/O participants and waits for all the persistence participants to complete.</span></span> <span data-ttu-id="e0c24-132">지속성 에피소드가 트랜잭션인 경우 이 트랜잭션은 Transaction.Current에서 제공됩니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-132">If the persistence episode is transactional, the transaction is provided in Transaction.Current.</span></span>

-   <span data-ttu-id="e0c24-141">다중 노드 시나리오에서 인스턴스는 다른 호스트에 의해 로드됩니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-141">In a multi-node scenario the instance is loaded by another host.</span></span>  <span data-ttu-id="e0c24-142">이러한 충돌 유형을 해결하는 방법은 몇 가지가 있습니다. 잠금을 소유하고 있는 노드에 처리를 전송하거나 다른 호스트가 해당 작업을 저장할 수 있도록 강제로 로드합니다.</span><span class="sxs-lookup"><span data-stu-id="e0c24-142">There are a few different ways to resolve these types of conflicts: forward the processing to the node which owns the lock and retry, or force the load which will cause the other host to be unable to save their work.</span></span>
