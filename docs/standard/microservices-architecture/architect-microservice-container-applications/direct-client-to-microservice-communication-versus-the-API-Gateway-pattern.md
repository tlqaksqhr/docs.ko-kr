## <a name="drawbacks-of-the-api-gateway-pattern"></a><span data-ttu-id="f55f1-244">API 게이트웨이 패턴의 단점</span><span class="sxs-lookup"><span data-stu-id="f55f1-244">Drawbacks of the API Gateway pattern</span></span>

- <span data-ttu-id="f55f1-245">가장 중요한 단점은 API 게이트웨이를 구현할 때 해당 계층을 내부 마이크로 서비스와 결합한다는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-245">The most important drawback is that when you implement an API Gateway, you're coupling that tier with the internal microservices.</span></span> <span data-ttu-id="f55f1-246">이와 같은 결합으로 인해 응용 프로그램에 심각한 문제가 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-246">Coupling like this might introduce serious difficulties for your application.</span></span> <span data-ttu-id="f55f1-247">Azure Service Bus 팀의 아키텍트인 Clemens Vaster는 GOTO 2016에서 자신의 “[메시징 및 마이크로 서비스](https://www.youtube.com/watch?v=rXi5CLjIQ9k)” 세션 중 이 잠재적 문제점을 “새로운 ESB”라고 언급합니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-247">Clemens Vaster, architect at the Azure Service Bus team, refers to this potential difficulty as “the new ESB” in the "[Messaging and Microservices](https://www.youtube.com/watch?v=rXi5CLjIQ9k)" session at GOTO 2016.</span></span>

- <span data-ttu-id="f55f1-249">API 게이트웨이는 추가 네트워크 호출로 인해 응답 시간 증가로 이어질 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-249">An API Gateway can introduce increased response time due to the additional network call.</span></span> <span data-ttu-id="f55f1-250">그러나 이 추가 호출은 일반적으로 내부 마이크로 서비스를 너무 빈번하게 직접 호출하는 클라이언트 인터페이스를 갖는 것보다는 적은 영향을 미칩니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-250">However, this extra call usually has less impact than having a client interface that is too chatty directly calling the internal microservices.</span></span>

- <span data-ttu-id="f55f1-256">API 게이트웨이가 단일 팀에서 개발된 경우 개발 병목 상태가 있을 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-256">If the API Gateway is developed by a single team, there can be a development bottleneck.</span></span> <span data-ttu-id="f55f1-257">이것이 다양한 클라이언트 요구 사항에 응답하는 세분화된 API 게이트웨이가 여러 개인 것이 더 나은 방법인 또 다른 이유입니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-257">This is another reason why a better approach is to have several fined-grained API Gateways that respond to different client needs.</span></span> <span data-ttu-id="f55f1-258">또한 내부 마이크로 서비스에서 작업 중인 다른 팀에서 소유하는 여러 영역 또는 레이어로 API 게이트웨이를 내부적으로 구분할 수도 있습니다.</span><span class="sxs-lookup"><span data-stu-id="f55f1-258">You could also segregate the API Gateway internally into multiple areas or layers that are owned by the different teams working on the internal microservices.</span></span>
